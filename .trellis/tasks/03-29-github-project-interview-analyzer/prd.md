## Problem Statement

MemCoach 当前已经具备简历模拟面试、专项训练、知识库检索、录音复盘与用户画像能力，但缺少一个围绕“真实项目经历”展开的输入源。用户现在只能基于简历或知识主题训练，不能直接把自己的 GitHub 项目交给系统，让系统围绕该项目生成深挖面试问题、项目拆解报告与项目经历草稿。

本次设计要解决的问题不是“让 MemCoach 读取任意开发环境”，而是把一个公开 GitHub 仓库转化为可用于项目面试准备的结构化材料。第一版目标是形成一个清晰、可复现、可展示的闭环：用户输入公开仓库 URL，选择分支，系统锁定 commit，执行一次临时代码快照分析，输出项目深挖问题与项目拆解结果。

## What Makes This Cool

- 用户不需要手工整理文档，直接把 GitHub 项目变成“项目面试包”。
- 分析对象不是模糊的“某个仓库”，而是明确到 `repo + branch + commit_sha` 的一次快照，结果可追溯、可复现。
- 这不是普通代码总结工具，而是把“项目理解”直接转化成“面试表达材料”，和 MemCoach 现有训练能力天然闭环。

## Constraints

- 第一版只支持公开 GitHub 仓库。
- 第一版入口为“粘贴公开仓库 URL”，不引入 GitHub OAuth 绑定。
- 系统默认预选仓库默认分支，用户可以手动切换到其他分支；一次分析任务只对应该分支当前解析出的一个 `commit_sha`。
- 用户在开始分析前必须补充“项目角色 / 负责模块或目录”的自声明，避免系统生成超出其实际负责范围的问题。
- 不做跨分支自动融合，不做“自动挑选最优分支”。
- 不做整仓库长期保存；源码仅用于临时分析，分析完成后删除。
- 服务器长期保留的是“精简分析包”，而不是完整仓库副本。
- 第一版采用异步分析任务 + 结果轮询，不在单个 HTTP 请求中阻塞完整分析链路。
- 每条“项目深挖问题”和关键结论都必须绑定证据来源（文件、配置、目录或依赖线索）。
- 第一版优先产物顺序为：`项目深挖问题与参考答题点 > 项目核心功能与模块拆解 > 项目经历草稿 > 知识盲点提示`。
- 第一版不实现“项目分析官对话”能力，该能力进入后续阶段。
- 第一版结果页必须提供“进入项目答题训练”的后续入口，形成 MemCoach 内部闭环。
- 需要尽量复用现有的用户隔离、索引、会话与前端交互能力，避免另起一套系统。

## Premises

以下前提已在本轮会话中收敛确认，后续设计默认基于这些前提推进：

1. 该功能同时服务两个目标：答辩展示价值与个人项目面试准备价值，但 Phase 1 的主 KPI 是“完成一次稳定、可信、可复现的项目面试包生成”，而不是覆盖所有真实仓库场景。
2. 第一版应该优先证明“项目面试官”能力，而不是先做完整 GitHub 账号体系。
3. “导入项目”的本质不是上传依赖与环境，而是获取足够用于理解项目的源码、关键配置、README 与少量仓库元信息；第一版不依赖 PR / Issue / 私有上下文。
4. 纯 GitHub API 浅读不够，长期完整 clone 也不合适；合理中间态是“API 预扫描 + 临时代码快照分析”。
5. 多分支问题必须被严格定义：一次分析只面向用户选定分支上的一个确定 commit。
6. 第一版的 3 分钟演示闭环应聚焦于“项目深挖问题 + 项目拆解报告 + 进入项目答题训练的下一步入口”，项目经历草稿可作为加分项而非主舞台。

## Approaches Considered

### Approach A: 纯 GitHub 接口浅读

Summary: 用户粘贴公开仓库 URL 后，系统仅通过 GitHub API 读取 README、语言分布、目录树与少量关键文件，不下载完整代码快照。  
Effort: S  
Risk: High  
Pros:
- 实现最快，服务端存储压力最小。
- 不需要处理临时目录清理和大批量源码扫描。
- 答辩时容易解释“轻量化分析”。
Cons:
- 很难做出高质量的跨文件项目深挖问题。
- 容易退化成“仓库摘要器”，不够像真正的项目面试官。
- 产出解释力弱，用户很难信任问题来源。
Reuses: 现有前端页面框架、用户体系、结果展示模式。

### Approach B: GitHub API 预扫描 + 分支级临时代码快照分析

Summary: 用户粘贴公开仓库 URL 后，系统先读取仓库元信息和分支列表，用户选择分支，后端把该分支解析为一个确定 commit，下载该 commit 的临时代码快照，完成分析后删除源码，仅保留精简分析包。  
Effort: M  
Risk: Medium  
Pros:
- 在分析质量、存储成本、可解释性之间取得最优平衡。
- 可以明确回答“多分支如何处理”，并且结果可复现。
- 能支持高质量的项目深挖问题与模块拆解。
Cons:
- 需要处理临时目录、快照下载、仓库大小上限与超时控制。
- 需要设计“精简分析包”格式与缓存策略。
- 对超大仓库和复杂 monorepo 仍需额外限制。
Reuses: `backend/indexer.py` 的文档索引思路、`backend/storage/sessions.py` 的结果持久化模式、现有前端训练/结果页面模式、现有用户隔离目录。

### Approach C: GitHub 账号绑定 + 仓库列表 + 项目分析官异步任务

Summary: 用户在 MemCoach 中绑定 GitHub 账号，从仓库列表中选择项目，系统以异步任务形式执行分析，并进一步扩展为可持续对话的项目分析官。  
Effort: L  
Risk: Medium  
Pros:
- 用户体验最好，长期产品形态最完整。
- 天然支持重新分析、历史记录、后续项目对话。
- 更像一个成熟产品，而不是单次 Demo。
Cons:
- OAuth、权限、token 管理、任务队列、历史数据模型都会明显扩展范围。
- 第一版很容易把时间花在接入和账户体系，而不是核心分析质量。
- 不适合作为毕设第一阶段的最小可展示闭环。
Reuses: 现有认证体系、会话持久化、未来可复用现有训练记录和历史页面。

## Recommended Approach

选择 **Approach B**。

原因很明确：它既避免了“纯 GitHub 浅读”的分析深度不足，也避免了“长期完整 clone”的存储与边界失控。对答辩和实际使用来说，`公开仓库 URL -> 选择分支 -> 锁定 commit -> 临时快照分析 -> 输出项目面试包` 是最稳、最好讲、也最容易被验证的一条线。

### 一期范围

#### 输入

- 用户粘贴公开 GitHub 仓库 URL。
- 系统解析仓库基本信息，默认预选默认分支，并展示可选分支。
- 用户可保持默认分支，也可切换到其他分支。
- 用户补充自己在该项目中的角色、负责模块或重点目录。
- 系统解析该分支当前头指针为一个确定 `commit_sha`。

#### 分析流程

1. 通过 GitHub API 获取仓库元信息、默认分支、分支列表、语言占比、README 与关键配置文件线索。
2. 创建持久化分析任务记录，状态从 `queued` 进入 `fetching / filtering / analyzing / completed / failed / cancelled`。
3. 下载指定 `commit_sha` 的源码快照到临时目录，并执行安全解压。
4. 执行文件过滤与切片，只保留源码、README、配置、依赖声明等高价值输入。
5. 由“项目分析官”结合用户自声明的负责范围完成结构理解与产物生成。
6. 删除临时源码目录。
7. 保留精简分析包，用于结果展示、下载和后续复看。
8. 结果页提供进入“项目答题训练”的入口，复用 MemCoach 现有训练能力。

#### 一期核心产物

1. 项目深挖面试问题
   - 每题附参考答题点
   - 每题附可继续追问的方向
   - 每题附证据来源（文件 / 配置 / 目录 / 依赖线索）
2. 项目拆解报告
   - 核心功能
   - 关键模块
   - 技术栈
   - 亮点与潜在难点
   - 与用户自声明负责范围的对应关系
3. 项目经历草稿
   - 面向简历/项目经历描述
   - 强调职责、方案、亮点、结果

#### 一期明确不做

- 私有仓库
- GitHub OAuth 绑定
- 跨分支聚合分析
- 自动推断“最值得分析的分支”
- 项目分析官对话
- PR / Issue / Commit 历史的深度分析
- Monorepo 子目录级选择
- 大规模多仓库管理

## Open Questions

## Success Criteria

- 用户可以在 Web 页面输入公开 GitHub 仓库 URL，并成功解析仓库信息。
- 用户可以看到仓库分支列表，默认分支已预选，且可手动切换后开始分析。
- 用户可以在分析前补充自己的项目角色与负责范围。
- 系统在结果页明确展示 `仓库 / 分支 / commit_sha / 分析时间`。
- 系统在分析过程中明确展示状态、失败原因与重试入口，而不是长时间无反馈。
- 一次分析可以稳定产出至少 5 条项目深挖问题，每条包含参考答题点。
- 每条项目深挖问题都能展示对应证据来源。
- 系统可以生成一份结构化项目拆解报告，至少覆盖核心功能、关键模块、技术栈三个维度。
- 系统可以生成一版项目经历草稿，供用户直接改写到简历中。
- 服务端不会长期保留完整源码目录，分析结束后仅保留精简分析包。
- 用户可以从结果页直接进入一次“项目答题训练”或等价练习入口。
- 对中等规模公开仓库，用户可以在可接受时间内拿到结果；第一版目标是“单次分析体验明显短于手工整理项目面试材料”。
- 该功能可以在答辩演示中通过一次真实仓库输入完成闭环展示。

## CEO Review

### 0A. Premise Challenge

1. `GitHub 仓库分析` 不是最终价值本身，真正价值是“把项目经历变成可用于面试表达和训练的材料”。
   结论：接受，但将 GitHub 定位为 Phase 1 的主要输入源，而不是长期唯一入口。
2. `答辩展示价值 + 真实面试准备价值` 可以共存，但不能没有主次。
   结论：接受，Phase 1 主 KPI 调整为“完成一次稳定、可信、可复现的项目面试包生成”，次级 KPI 是“支持进入训练”。
3. `只看源码和配置就足够` 这个前提过强。
   结论：修正为“源码 + README + 关键配置 + 仓库元信息 + 用户自声明负责范围”足以支撑 Phase 1；PR / Issue / commit 历史延后。
4. `用户必须自己选分支` 过于工程导向。
   结论：修正为默认预选默认分支，允许高级用户切换。
5. `生成静态报告即可` 与 MemCoach 主线不匹配。
   结论：修正为结果页必须有进入训练的入口，否则闭环不完整。
6. `结果只要看起来合理即可` 不可接受。
   结论：修正为每条问题和关键结论必须带证据来源，否则无法建立信任。

#### 状态设计

| 状态 | 必须展示 |
|---|---|
| Empty | 输入规则、只支持公开仓库、体积限制 |
| Validating | 仓库解析中 |
| Branch loaded | 默认分支、可切换分支、负责范围表单 |
| Queued / Running | 当前阶段、进度说明、取消入口 |
| Partial failure | 哪一步失败、是否可重试 |
| Completed | 结果工作区 + 训练入口 |

1. 不做“一个空页面 + 一个开始按钮”的模糊设计，必须是短向导。
2. 默认分支预选，避免把工程概念变成首要用户负担。
3. `负责范围声明` 是信任控件，不是可选附加项。
4. 结果页第一屏先给“这个项目是什么 + 你负责什么 + 现在可以做什么”，不是直接堆长报告。
5. 每条问题旁边展示证据入口，否则结果不可解释。
6. 结果页必须有明显的“开始项目答题训练”按钮，而不是把用户丢回首页。

#### Responsive / Accessibility

- 移动端结果页使用分段 Tab，不在首屏堆完整长报告。
- 训练入口始终可见，避免滚动后丢失下一步。
- 关键按钮满足键盘可达性与清晰焦点态。

## Engineering Review

### Scope Challenge

该方案不能直接复用 `sessions` 表硬塞项目分析结果。现有 `sessions` 语义是“面试/训练会话”，包含 transcript、scores、review。项目分析是另一类对象，生命周期、状态机、结果结构和失败路径都不同。为了保持最小改动但不污染现有模型，建议新增独立的 `project_analyses` 存储模块，而不是复用 `sessions`。

另一个关键点是：现有 `startInterview -> navigate -> immediate render` 是短时请求模式，但仓库分析明显是长任务。如果继续套用同步请求，云端部署会在超时、断网、GitHub 限速、LLM 波动下直接变脆。Phase 1 应采用“创建任务 -> 轮询状态 -> 渲染结果”的明确模型。

### Performance / Security / Reliability

#### Performance

- 优先下载指定 `commit_sha` 的 archive，而不是完整 clone。
- 配置化限制仓库总大小、文本文件数、单文件大小。
- 先做元信息预扫描，再做重分析，避免无意义下载。
- 结果按 `repo + branch + commit_sha + owned_scope_hash` 做用户级缓存。

#### Security

- 只接受 `github.com` 公开仓库 URL。
- archive 解压必须防止路径穿越。
- 不执行仓库内任何代码，不安装依赖，不运行脚本。
- 对输出的 evidence 只保留必要片段，不长期保留完整源码。

#### Reliability

- 状态机必须可恢复到 `failed`，不能无限 `analyzing`。
- 结果页必须显示“分析基于哪个 commit”。
- GitHub 和 LLM 失败都必须是可解释错误，而不是空白页。

## Cross-Phase Themes

1. **证据溯源是高置信主题**

4. **必须采用持久化任务状态，而不是同步阻塞请求**  
   这是架构、设计和上线稳定性共同要求。
